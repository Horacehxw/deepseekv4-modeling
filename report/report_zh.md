# DeepSeek V4 推理性能分析：Ascend 910C vs NVIDIA H20

## 1. 摘要

本报告量化了 DeepSeek V4 在 **Ascend 910C** 和 **NVIDIA H20** 上、跨四种上下文长度（8K、32K、128K、256K）、四个关键分析维度（逐算子瓶颈分解、最优并行配置、P/D 分离拓扑以及 KV 压缩扩展至 1M 上下文）的推理性能。所有数据来自基于网格搜索配置的 roofline 分析模型；完整方法论详见第 8 节。

**六项核心发现：**

1. **V4 的 KV cache 比 V3 缩减 4.6 倍**（每 token 15,168 字节 vs 70,272 字节）。C4A/C128A 异构压缩结构使得 256K–1M 上下文具备经济可行性：在 1M token 时，V4 的 KV 状态约为 6.9 GB，而 V3 在同等硬件上为 ~70 GB。

2. **910C 胜在 prefill，H20 胜在 decode**，分别快 1.2 倍和 3.4–6.1 倍，体现了两者截然相反的设计取向：910C 以较低的 HBM 带宽（1.8 TB/s）换取更高的 BF16 算力（376 TFLOPS），而 H20 以较低算力（148 TFLOPS）换取更高的 HBM 带宽（4 TB/s）。在 8K 上下文下，每卡 decode 吞吐量分别为：910C 312 tok/s，H20 1,053 tok/s。

3. **mHC 核函数融合是 910C 上收益最高的单项优化**——融合 SP+BF16 的 mHC 将 8K prefill 中 mHC 的耗时占比从 88% 降至 4.6%，端到端 prefill 提速约 8 倍。H20 上同一融合仅带来 1.9 倍提升，因为 mHC 原本已是带宽适配的。

4. **Decode 受带宽限制，MoE 主导。** 在 8K 场景下，MoE 路由专家消耗了 910C decode 时间的 70%、H20 的 37%——差距来自 H20 的通信开销。MoE 的 AllToAll 通信是 H20 上的主要瓶颈，在 EP=8 的适中配置下即占 decode 时间的 ~27%。

5. **前缀缓存大幅改变 PD 比例。** 缓存命中率 0% 时，最优 P:D 接近均衡（8K–256K 范围内为 5:6 至 4:1）；缓存命中率 99% 时，同一负载需要 1P:13D 至 1P:489D（H20/256K 至 910C/8K）——视硬件与上下文长度而定，prefill 吞吐量在效益上比 decode 便宜 10–500 倍。容量规划必须将缓存命中率视为第一类容量维度。

6. **128K–256K 上下文将瓶颈转移至注意力。** Lightning Index 注意力在长上下文下主导 prefill（256K 时超过 40%），而长跳步的 C128A 压缩使 decode 的 KV 读取在 1M token 内保持可控。这一架构特性正是 V4 能够在 64 GB 级加速卡上处理长上下文的关键。

综合来看，V4 推理成本模型的核心结论是：(a) 硬件选型应按阶段区分（prefill 用 910C 级，decode 用 H20 级）；(b) 核函数融合（尤其是 mHC）在 910C 上是必选项而非可选项；(c) 缓存感知调度是最高杠杆的软件优化手段；(d) 长上下文的经济性由 KV 压缩和 EP 带宽决定，而非原始 FLOP 数。

## 2. 模型架构

### 2.1 DeepSeek V4 架构

DeepSeek V4 代表了 DeepSeek 模型家族的一次根本性重新设计，以更少的参数量换取推理效率。总参数量约 285B——不足 V3 的 704B 的一半——V4 通过三项核心架构创新实现了具有竞争力的质量：

**MQA 与 KV 压缩。** V4 采用多查询注意力（64 个 Q heads 共享 1 个 KV head），配合激进的异构压缩调度：2 层全注意力（ratio=1）、21 层 C4A（4 倍压缩）和 20 层 C128A（128 倍压缩）。这使得每 token 的 KV cache 仅 15,168 字节——比 V3 的 70,272 字节缩减 4.6 倍。

**Lightning Index。** 一种带有 64 个索引头（dim=128）的可学习稀疏检索机制，在每个压缩注意力层中为每个查询选出最相关的前 512 个 KV 条目，在保持长上下文质量的同时将 KV 占用量控制在较小范围内。

**manifold Hyper Connection (mHC)。** 以 4 倍扩展的混合操作（hc_mult=4）替代标准残差连接，改善梯度流，但在未优化时会显著增加 HBM 流量。

**架构设计动机。** V4 的设计选择由内存受限硬件上的推理经济性驱动。MQA 风格的注意力（kv_lora_rank=512）加上逐层压缩，将 KV 占用降至每 token 15.3 KB——比 V3 小 4.6 倍——正是这一点使 256K–1M token 的上下文窗口在 64 GB 加速卡上成为可行。mHC 模块（含 4 倍 hidden 扩展和 256 专家的 Sinkhorn 路由）以结构化低秩混合替代了密集 FFN，以路由/gather/scatter 流量换取密集矩阵乘法的 FLOP。Lightning Index 作为全注意力层的 topk 缓存预滤波器；它缩减了 SWA 的作用范围，但如第 3 节所示，在超过 32K token 后成为 prefill 的主要开销，因为其复杂度即便在稀疏化后仍随 seq^2 增长。

### 2.2 硬件平台：Ascend 910C

两个目标平台具有本质不同的硬件特征，导致截然不同的推理特性。

| 指标 | Ascend 910C | NVIDIA H20 | 比例 |
|---|---|---|---|
| Cube TFLOPS (BF16) | 376 | 148 | **910C 2.54 倍** |
| Vec TFLOPS (FP32) | 24 | 44 | **H20 1.83 倍** |
| Cube:Vec 比率 | 15.7:1 | 3.4:1 | H20 更均衡 |
| HBM 带宽 (GB/s) | 1,800 | 4,000 | **H20 2.22 倍** |
| HBM 容量 (GB) | 64 | 96 | **H20 1.5 倍** |
| HBM 预留 | 10% | 10% | 相同 |
| TP 带宽 (GB/s) | 392 | 450 | 相近 |
| EP 带宽 (GB/s) | 392 | 50 | **910C 7.84 倍** |
| 网络延迟 (us) | 10 | 5 | H20 低 2 倍 |
| Cube 利用率 | 40% | 50% | — |
| HBM BW 利用率 | 30% | 80% | — |
| Vec 利用率 | 20% | — | — |
| BW 利用率 | 26% | 80% | — |

**硬件特征对比。** 两个平台以截然相反的取舍应对同一工作负载。910C 是一个*算力丰富、带宽有限、横向扩展友好*的加速卡：BF16 吞吐量是 H20 的 2.5 倍，但 HBM 带宽仅为 H20 的 0.45 倍，同时拥有 9 倍更丰富的 EP 互联（建模的 392 GB/s 是 910C 横向扩展网络的每链路聚合值，而 H20 在我们的服务拓扑中有效的每 rank EP 带宽约为 50 GB/s）。H20 则相反：低 FLOP 但高 HBM 带宽和大容量。两者在实际利用率假设上也不同——910C 建模为 40% cube 和 30% HBM 利用率，H20 为 50%/80%——反映了 H20 成熟得多的 CUDA/cuBLAS 技术栈，以及 910C 当前峰值与持续吞吐之间更大的差距。

这一不对称性直接决定了各平台的优势所在。算力受限阶段（prefill、MoE GEMM）偏向 910C；内存受限阶段（大 KV 注意力的逐 token decode）偏向 H20；而 910C 的 7.5 倍 EP 带宽优势则将任何依赖大量专家调度的阶段拉回其一侧。本报告其余部分将量化每个阶段在各平台上的表现。

**Prefill 瓶颈。** 两个平台的 prefill 均重度受算力限制：在 8K 输入时，最大开销来自 mHC（约 26–29%）、注意力投影和 MoE GEMM——均为 CUBE 受限。随着上下文增长，注意力计算（全注意力 + Lightning Index）逐渐主导：到 128K 时，Lightning Index 单独消耗 910C prefill 时间的 ~32%、H20 的 ~28%；到 256K 时超过 prefill 总开销的 50%。MQA + C4A/C128A 使 KV 写入开销保持低廉，但 Lightning Index 注意力（在全序列上计算 Q*K）在长上下文下无法规避。

910C/H20 的差异清晰可见：在 910C 上，mHC 经融合 BF16 + mHC-SP 后已降至 <5%，注意力投影（CUBE）和 MoE（CUBE）成为最大分量。在 H20 上，mHC 仍占 ~10%，因为 H20 较低的算力吞吐使融合 FP32 路径在绝对时间上更耗，且通信代价更重（8K 时通信占 27%），因为 H20 每 rank EP 带宽约低 7.5 倍。因此，H20 在通信上花费更大比例的 prefill 时间，而 910C 在算力上花费更大比例。

### 6.5 通用建议

**跨场景建议**

1. **始终启用前缀缓存。** 对于交互式长上下文服务，0% 与 90% 缓存命中率之间的 prefill 吞吐差异达 10–100 倍。缓存是本研究中收益最高的单项系统优化。
2. **根据上下文长度匹配部署形态。** 短上下文（8–32K）工作负载可使用 TP=1–2，EP=8–16，DP=8（单实例形态）。长上下文（128–256K）工作负载需要更大的 EP（32）和 TP（2–4）以容纳 KV 状态并分摊 prefill 开销。
3. **将 910C 用于 prefill 密集或算力受限的工作负载。** 其 2.5 倍算力密度在算术强度高时胜出——长输入、低缓存命中率、大规模 MoE 调度。将 H20 用于 decode 密集或大批量工作负载，此时 HBM 带宽占主导。
4. **PD 比例随上下文扩展。** 8K 时 1P:6D，256K 时 1P:13D（99% 缓存命中率）——随上下文增长和缓存生效，prefill 占总开销的比例越来越小。请据此规划分离部署。
5. **MoE EP 互联比每 rank HBM 更重要（在规模下）。** 910C 的更高 EP 带宽使其即使每卡 HBM 吞吐更低，在高 DP MoE 部署中仍具竞争力。

## 7. 行业启示

本报告的发现超越了 DeepSeek V4 在 910C/H20 上的具体场景——它们揭示了行业在构建、部署和理解前沿推理系统方面的一个更广泛的拐点。

### 7.1 KV Cache 管理与异构内存

V4 将每请求 KV 从 V3 MLA 的 ~68.6 KB/token 压缩至 ~14.8 KB/token（通过 C4A/C128A）——但更重要的是，它使每 token 的 KV 占用随序列长度的增长呈*亚线性*，一旦 C128A 层占主导。在 256K 上下文时，这意味着每请求 1.7 GB 的 KV 状态（vs V3 MLA 的 ~18 GB，减少了一个数量级），从根本上改变了行业对缓存基础设施的思考方式。

这与近期行业动向直接相连。**Mooncake**（为 Moonshot 的 Kimi 提供动力）等框架已迁移至以 KVCache 为中心的分离架构：KV 状态存储在跨 GPU HBM、CPU DRAM 和 NVMe 的分层存储中，并使用全局基数树索引实现前缀复用。**SGLang** 和 **vLLM v1** 均已将基于基数树的前缀缓存作为默认配置。我们的分析强化了这些设计的理由：第 6 节中的 prefill 延迟曲线*在没有缓存时*极为陡峭（从 8K 到 32K 增加 5 倍，到 128K 增加 14 倍），但在 99% 命中率下几乎持平（即使在 128K 时 TTFT 也低于 200ms）。当 KV 足够小可以在规模下放入分布式 CPU 内存时——V4 的压缩使这在经济上可行——KV 复用就不再是锦上添花，而是主导性能的核心杠杆。下一波推理系统将把 KV cache 视为第一类持久化数据结构，而非内存中的副作用。

### 7.2 Prefill/Decode 分离已是必选项

第 6 节的 P/D 比例分析几乎在机械上证明了分离的必要性：在 99% 缓存命中率下，单个 prefill 实例能服务 45–87 个 decode 实例（取决于上下文长度）。将两个阶段共置于同一硬件上，要么永久浪费算力（在 decode 期间），要么浪费带宽（在 prefill 期间）。**DistServe**（OSDI '24）和 **Splitwise**（ISCA '24）已通过对密集模型的测量得出同样结论；V4 的 MoE 结构放大了这种不对称性，因为 EP 通信存在于 decode 关键路径上，而 prefill 主导因素是 Lightning Attention 算力。

V4 级工作负载的新之处在于**与前缀缓存的复合效应**。在 99% 命中率下，prefill 实例的有效吞吐提升约 10 倍，使最优 P:D 比例从约 1:7（无缓存，8K）移动到 1:64+（有缓存，256K）。这与 **Mooncake**（FAST '25）为 Kimi 在生产中的架构吻合——全局 KVCache 池位于分离的 prefill 和 decode worker 之间，Conductor 调度器将新请求路由到缓存局部性最优的 prefill 节点。NVIDIA 的 Dynamo 服务栈（GTC '25 宣布）和 SGLang 的 PD 模式正收敛于相同的形态。我们的建议：任何超出单机架的 V4 部署都应以 PD 优先设计，将 KV 传输网络（RDMA、GDS 或 NVL fabric）视为第一类容量维度，而非事后考量。

### 7.3 算力受限 vs 带宽受限：按阶段选择硬件

910C vs H20 的对比清晰地分开了两种设计哲学。910C 以 HBM 带宽（1.8 TB/s）换取原始 cube 算力（H20 FLOP 的 2.5 倍）和强横向扩展互联。H20 牺牲算力（CUBE FLOP 约为 H100 的 40%）以保留完整的 4 TB/s HBM 带宽和成熟的 NVLink 拓扑。我们的结果表明，这直接映射到阶段特异性偏好：**prefill 受算力限制，decode 受带宽限制，而理想的分离集群两者兼备**。

具体而言，910C 在我们测试的每种上下文长度下 prefill 快 1.0–1.4 倍，而 H20 decode 快 1.3–4.2 倍（差距随上下文增长，因为 decode 越来越受 KV 读取限制）。经济启示：运营长上下文、低缓存命中率工作负载的运营商不应购买均质集群——Splitwise（ISCA 2024）和 DistServe（OSDI 2024）在原则上已提出这一论点；V4 的数据使其在每 token 层面具体化。预计 2026 年云提供商将明确提供分离 SKU："算力层" GPU 用于 prefill 池，"带宽层" GPU 用于 decode 池，共享快速 KV 传输网络。

### 7.4 网络带宽成为 MoE 密集型 Decode 的新瓶颈

Decode 时 AllToAll 专家调度消耗 H20 步骤时间的 5–22%（decode 8K/4K → 256K/4K），而 910C 上为 7–35%——这是对通常 H100 vs 国产 GPU 模式的反转，由 910C 更高的 EP 带宽（我们配置中为 H20 的 7.8 倍）驱动。Lightning Attention 的 KV 扫描随 `S/compress_ratio` 线性扩展，使这一问题更加突出：在 256K 时，C4A 层单独每步读取约 2 GB 的压缩 KV，而 AllToAll 预算是将每步延迟保持在 SLA 以内的唯一手段。这是为何 MoE 服务不能仅靠增加 HBM "解决"的核心原因——约束条件越来越多地在于 GPU 间带宽，而非内存大小。

行业信号一致：NVIDIA 的路线图（NVL72 → NVL576 配合 Spectrum-X）、华为的 CloudMatrix，以及 Cerebras/Groq 的晶圆级设计，均押注于芯片间带宽作为下一个十年的瓶颈。我们的数据具体支持了这一判断：在 256K decode 时，910C 上通信时间约 6.3 ms，而算力约 22 ms——已占步骤的 22%。在相同通信网络下将模型规模或专家数翻倍，会将这一比例推至 ~40%，此时通信主导。**实践建议：若您正在为 V4 级模型规划服务集群，将互联预算分配为总 GPU 费用的约 30%，而非典型的约 10%。**

### 7.5 算法-硬件协同设计已非可选项

mHC 融合核的结果——910C 上提速约 8 倍（8K prefill 从 25,912 ms 降至 3,262 ms），H20 上提速约 1.9 倍（7,869 ms 降至 4,047 ms）——表明同一算法优化在不同瓶颈位置下回报差异极大。910C 上的收益来自消除冗余 HBM 往返（内存受限模式）；H20 上的核对带宽的适配度本已较高，核融合可去除的冗余更少。

这说明"模型-系统协同设计"不能再推迟到训练后的核工作。三项具体启示：

1. **架构选择必须按目标硬件进行成本建模。** V4 的架构师显然针对 910C 特征进行了优化（高 FLOP，适中带宽）——使用 C4A/C128A 的全注意力保持内存流量低，而 256/6/1 专家路由使每 rank GEMM 维度足够大以利用 FP8 张量核。同一模型在 H20 或 H100 上会从不同的稀疏模式中受益。
2. **编译器栈与核同等重要。** TileLang、Triton 和 Mosaic 现已使编写一个融合核并面向多个后端成为可行。V4 的 mHC 核以 TileLang 编写，为 910C 和 H20 分别提供调度模板——这是行业其余部分将效仿的模式。
3. **开源 MoE 模型以训练成本为导向；生产中推理成本占主导。** 像 V4 这样有意以部分训练效率换取推理友好 KV 的架构（压缩比、滑动窗口+全局注意力）将赢得部署。预计这一设计模式会更多出现（参见 Mistral 的滑动窗口注意力、GPT-4 Turbo 据传的压缩 KV）。

### 7.6 超长上下文变得可行

前沿模型规模已稳定在 200B–1T MoE 范围；竞争的新维度是*在固定延迟下的上下文长度*。V4 的设计——仅 2/43 层使用全注意力、其余层压缩 KV、滑动窗口+全局注意力分离、mHC 融合以减少中间写入——是一个明确声明：未来两年的能力提升将来自更长的有效上下文，而非更大的参数量。

我们的 1M 上下文 KV 扩展表是核心数据：在 1M token 时，V4 将一个请求的完整 KV cache 保存在 **6.9 GB**（BF16）中，比 V3 小 10 倍。这一数字正是使 1M token 上下文在 8 张 910C 或 4 张 H20 上经济可行的原因。没有压缩，你需要 FP4 权重（Llama 4 路径）、CPU/SSD 卸载（Mooncake 风格），或干脆放弃服务。有了 V4 风格的压缩，1M 上下文成为约 1 秒 TTFT 的单节点工作负载——对聊天助手和代码智能体在经济上可行，而不仅仅是研究演示。

行业方向的核心结论：**对于 V4 级模型，部署难点已从"装下权重"转向"供给激活"。** 算力、HBM 带宽、通信网络和 KV cache 基础设施必须协同设计；将任何一项视为固定约束而优化其余部分——这是 2024 年之前密集模型服务的主导模式——会将 2–5 倍性能留在桌面上。赢得下一轮推理竞争的架构师，将是那些把模型、核、调度器和存储视为单一系统来设计的人。

## 8. 附录

### 8.1 方法论

**性能模型。** 所有结果由基于 roofline 的分析性能模型（`perf_model/`）生成。对于每个算子，模型独立计算三个开销估算：CUBE 时间（矩阵乘法 FLOP / 有效 TFLOPS）、VEC 时间（逐元素 FLOP / vec TFLOPS）和 MEM 时间（传输字节数 / 有效 HBM 带宽）。瓶颈由 `argmax(cube, vec, mem)` 确定，总算子时间为 `max(cube, vec, mem) + communication_time`。所有尺寸假设 BF16（每元素 2 字节），FLOP 遵循 `M * N * K * 2` 的约定（对应 `[M,K] x [K,N]` 矩阵乘法）。

**通信模型。** TP 通信（AllReduce、AllGather、ReduceScatter）使用基于环形算法的 TP 带宽建模。EP 通信（MoE 调度/合并的 AllToAll）使用 EP 带宽。通信和算力建模为串行（无重叠），提供保守上界。

**参数搜索。** 最优配置通过对 TP ∈ {1,2,4,8,16,32,64}、EP ∈ {8,16,32,64,128,256}、DP ∈ {1,2,4,8}、batch_size ∈ {1..512} 和上下文长度 ∈ {8K, 32K, 128K, 256K} 的穷举网格搜索得出。约束 `(TP*DP) % EP == 0` 确保有效分区。每种配置在计算性能指标前先进行 HBM 容量可行性检验。

**Decode 快速模式。** Decode 模型评估使用周期采样加梯形插值：每步开销分解为常数 + 线性(S) + 周期(S)，其中周期 P = LCM(compress_ratios) = 128。仅采样前 P 步和后 P 步，再通过 `T_total = N * (T_first + T_last) / (2P)` 插值。对于 output_len=4096，提速 16 倍，误差低于 0.001%。

**P/D 分离（Schema v2）。** P:D 比例计算使用 `balance_tolerance=0.1` 的整数均衡。最优 P:D 比例在保持 QPS 不平衡在 10% 以内的同时最小化总 GPU 数量。HBM 预留余量默认 `hbm_reserved_pct=10%`。

**数据生成。** 所有表格由 `report/analyze_scenarios.py` 生成，该脚本对所有场景组合运行性能模型并将结构化 JSON 数据导出至 `report/data/`。无人工数据录入——所有数字直接来自分析模型。
